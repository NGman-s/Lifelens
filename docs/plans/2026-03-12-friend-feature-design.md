# 好友功能设计文档 (Friend Feature Design)

## 1. 方案选型 (Approach)
选择 **方案C：全场景混合模式（二维码 + 口令双管齐下）**。
- **免注册体验**：前端生成 UUID 作为身份标识，后端轻量化存储。
- **全场景互联**：每位用户同时拥有唯一的 6 位“短口令”和对应的识别二维码。
- **结合主业务**：前端利用既有的相机流与扫码组件（或提供文字输入接口）实现面对面扫码或远程复制短口令加好友。

## 2. 架构与数据流 (Architecture & Data Flow)
### 后端 (Backend - FastAPI + SQLite)
- **数据库设计**：引入 Python 内置的 `sqlite3` 建立轻量级本地文件数据库 `lifelens.db`，降低运维成本。
    - `users`: 存储 UUID、6位口令、注册时间。
    - `friendships`: 存储好友关系，双向。
    - `diet_records`: 记录用户的 UUID、分析出的基本饮食数据（菜名、热量、识别结果图片 URL）及记录时间。
- **新增接口**：
    - `POST /api/v1/user/init`: 根据用户提供的 UUID 或为空时下发新的 UUID 及口令。
    - `POST /api/v1/friends/add`: 通过 6 位口令或目标 UUID 加好友。
    - `GET /api/v1/friends/feed`: 获取该用户所有好友的“今日饮食”信息流。
- **修改已有接口**：
    - `POST /api/v1/vision/analyze`: 在分析成功、存下缩略图后，额外将此次识别的梗概保存到 `diet_records` 表。

### 前端 (Frontend - Uni-app)
- **全局状态 (Pinia)**：新增 `userStore` 保存 UUID 和自身口令，启动时验证或获取 UUID。向原生请求及 H5 请求中注入 UUID 头或参数。
- **UI 组件与页面**：
    - 新建好友模块/抽屉栏 (`pages/friends` 或独立组件)。
    - **“我的身份卡” (Glassmorphism 风格)**：展示自身的二维码和口令。
    - **“添加好友”弹窗/入口**：支持拉起设备扫码能力 (`uni.scanCode` 或相机组件扩展) 和手动输入口令。
    - **“好友动态”列表**：卡片式流式布局，展示好友今天吃了啥（基于返回的缩略图 `image_url`、名称 `main_name` 和热量）。

## 3. 安全与体验权衡 (Trade-offs)
- 由于无密码，若设备更换或清空缓存，用户可能丢失记录（可以通过复制 UUID 保存解决，但不强制）。
- 验证流程：没有严格的“好友申请通过”流程（考虑到熟人且“低阻力”特性，目前设计为单向或自动双向绑定，扫码即加）。
- 历史图片：原业务已包含 30 天自动清理机制，`diet_records` 内保存的也是缩略图地址并随清理脚本一并回收，存储压力极小。

## 4. 测试与验证 (Testing)
- 后端：在 `test_api.py` 增加一套初始化用户、加好友、查询信息流的 smoke test 流程。
- 前端：H5 和微信开发者工具验证 `uni.scanCode` / 口令输入页和组件展示效果。
